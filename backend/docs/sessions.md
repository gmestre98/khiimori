# Sessions & authentication

After a successful Google sign-in (see [oauth-signin.md](oauth-signin.md)) the
backend issues an authenticated **session**, validates it on every request
through shared **auth middleware**, and clears it on sign-out. Implemented across
[Epic M02.3](../../docs/khiimori-v1/milestones/M02-auth-and-profile/epic-03-sessions-auth-middleware/README.md).

## Mechanism: stateless signed cookie

The session is a **stateless, HMAC-signed httpOnly cookie** — there is no
server-side session store.

- **Why stateless.** The service is a scale-to-zero modular monolith that may run
  several Cloud Run instances and is torn down when idle (PRD §7.0, §8.6). A
  signed cookie survives instance churn, needs no shared store, and adds no
  per-request database read. This mirrors the OAuth state cookie
  ([`state.go`](../internal/auth/state.go)).
- **Cookie value:** `base64url(user_id).<issued_unix>.<expires_unix>.<hmac>`,
  where the HMAC-SHA256 (keyed by `SESSION_SECRET`) authenticates the first three
  fields. The user id is base64url-encoded so it can never contain the `.`
  separator. Validation checks the MAC in constant time **before** trusting any
  field, then enforces expiry.
- **Contents:** only the user id (plus timestamps). "Who you are." Authorization
  ("what you may touch") is layered on in Milestone 08, which reads the same
  principal the middleware attaches (`internal/platform/authn`).

The mechanism lives behind a small interface ([`sessionManager`](../internal/auth/session.go)):
issuance (S1), the validating middleware (`RequireAuth`, S2), and sign-out (S3)
all go through it, so the mechanism can change without touching callers.

## Cookie attributes

| Attribute | Value | Why |
|-----------|-------|-----|
| `HttpOnly` | always | JS can't read the credential (XSS can't exfiltrate it). |
| `Secure` | prod only | required for `SameSite=None`; dev is plain-http localhost. |
| `SameSite` | `None` in prod, `Lax` in dev | prod: the web app (Firebase Hosting) and API (Cloud Run) are **cross-site**, so the SPA's `fetch` only sends the cookie with `None`. dev: both on localhost (same-site), so `Lax` works without `Secure`. |
| `Path` | `/` | sent on every API call so the middleware can protect any route. |
| `Max-Age` | 30 days | see expiry below. |

For the browser to send the cookie cross-origin, the API also returns
`Access-Control-Allow-Credentials: true` paired with the exact echoed origin
(never `*`) — see [`httpx/cors.go`](../internal/platform/httpx/cors.go).

## Expiry & sliding refresh

- **Lifetime: 30 days.** Long by design — re-auth must be smooth, not a hard
  logout mid-trip (PRD §6).
- **Sliding refresh.** On any authenticated request, once the session is past
  **half** its lifetime the middleware re-issues a fresh cookie
  (`refreshIfStale`). So an actively-used session effectively never expires,
  while **30 days of inactivity** lapses it. No separate refresh token or
  endpoint is needed — the slide is the refresh.

## The signing key (`SESSION_SECRET`)

- Read from config ([`SESSION_SECRET`](../internal/platform/config/config.go)).
  In production it is **auto-generated by the infra** (`@pulumi/random`) and
  stored in **Secret Manager**, injected as an env var on Cloud Run
  ([`infra/secrets.ts`](../../infra/secrets.ts), [`infra/cloudRun.ts`](../../infra/cloudRun.ts)) —
  no operator action, nothing in git. Locally, set it in `backend/.env`.
- **Never logged.** The key lives only in config and the HMAC; the shared logger
  also redacts any attribute whose key contains `secret`/`token`/`cookie`
  (M01.7), so an accidental structured log can't leak it.
- **Rotation.** Replacing the key invalidates **all** existing sessions — they
  fail the HMAC check and users simply sign in again. In prod, rotate by bumping
  the `keepers` on the `random.RandomBytes` resource (a new version is minted and
  mounted on the next deploy).

## Sign-out

`POST /auth/logout` clears the cookie (`Max-Age=-1`). Because the session is
stateless, sign-out is **client-side**: the browser drops the cookie and the
middleware then rejects the request as unauthenticated. It is public and
idempotent, so a stale/expired cookie can always be cleared.

**Trade-off:** there is **no server-side revocation list** in v1. A cookie
captured before sign-out would remain valid until its expiry. This is an
accepted trade-off for a single-user personal app (PRD §7.0 — minimal); the
documented upgrade path, if revocation is ever needed, is a per-user
session-version column bumped on sign-out (invalidating that user's live
cookies) or a short-lived-token + refresh design.

## CSRF stance

The session cookie is `SameSite=None` in prod (required for the cross-site SPA),
which by itself would expose state-changing endpoints to CSRF. This is mitigated
by the **JSON-API + CORS** shape: real state-changing endpoints accept/return
JSON, so a browser must send `Content-Type: application/json`, which is **not** a
CORS "simple request" and triggers a preflight — and the preflight only succeeds
for the configured web origins (`httpx/cors.go`). A cross-site attacker's page
therefore can't make a credentialed state-changing call.

The one simple-request exception is `POST /auth/logout` (no body): a cross-site
page could force a sign-out, whose only effect is clearing the victim's own
cookie — benign. New state-changing endpoints must keep to the JSON contract (or
add an explicit CSRF token) to stay protected.
